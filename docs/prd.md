# Vibe Tracing (简称VT) PRD

## 1. 项目背景

### 1.1 背景说明

在 Vibe Coding 场景中，非开发背景的人类通常不具备阅读代码、理解代码结构、判断实现质量、识别测试覆盖和审查代码风险的能力。

随着 AI Coding Agent 的能力增强，人类越来越多地从“直接写代码”转向“提出目标、约束方向、验收结果”。但这也带来了新的问题：  
AI Agent 可以快速生成代码、测试和说明文档，但人类未必能够判断这些产物是否真正可信。

传统开发流程中，需求、架构、任务、代码、测试、审查、合并通常由不同角色共同完成和制衡。但在 Vibe Coding 中，多个环节可能都由 AI Agent 执行，人类如果仍然依赖“读代码”来判断交付质量，就会失去有效治理能力。

因此，本项目希望建立一个面向 AI Coding 全生命周期的追踪与治理系统，把代码、测试、任务、Agent Claim、Review 结果等技术信息，转化为业务人员和项目管理者能够理解的治理信息。

---

### 1.2 当前问题

当前 Vibe Coding 过程中存在以下问题：

1. **非开发背景用户无法读懂代码**

   用户无法通过阅读代码判断：

   - 需求是否被正确实现
   - 数据是否被正确流转
   - 异常是否被正确处理
   - 边界条件是否被覆盖
   - 测试是否真的验证了验收标准
   - AI Agent 是否遗漏了关键逻辑

2. **AI Agent 的完成声明缺少可信证据**

   Coding Agent 可能声称任务已经完成，但实际可能存在：

   - 只实现了 happy path
   - 没有覆盖异常场景
   - 没有测试证据
   - 测试没有覆盖验收标准
   - 代码实现与需求不一致
   - 声称完成但没有可追踪 evidence

3. **需求、任务、代码、测试之间缺少追踪链路**

   目前很难直接回答：

   - 每个需求是否有任务覆盖？
   - 每个任务是否真的被实现？
   - 每个验收标准是否有测试？
   - 每个 Agent Claim 是否有证据支持？
   - 架构约束是否被违反？
   - 当前结果是否允许合并？

4. **项目管理者缺少面向 AI Coding 的验收抓手**

   在 AI 生成代码后，项目经理往往只能依赖 AI 的总结或少量运行结果进行判断。  
   这会导致项目管理过程缺少：

   - 证据链
   - 风险识别
   - 门禁判断
   - 可视化追踪
   - 可复核依据

5. **业务人员无法判断系统逻辑是否符合业务**

   业务人员关心的是：

   - 流程是否完整
   - 数据是否正确流转
   - 异常路径是否符合业务预期
   - 人工确认点是否合理
   - 风险是否可接受

   但 AI Coding 的交付物通常是代码、测试、技术说明，难以直接支持业务人员验收。

---

### 1.3 项目目标

通过本项目，让人类脱离“读代码”等开发技术栈，而聚焦在：

- 验收逻辑
- 识别风险
- 定义边界
- 拆解复杂系统
- 判断业务流程是否成立
- 判断数据流转是否正确
- 判断异常处理是否完整
- 判断 AI Agent 的结论是否有证据

本项目的核心目标是建立一套面向 Vibe Coding 的治理系统，让非开发背景的人类也能管理 AI Agent 的开发过程。

全生命周期治理的核心链条为：

```text
Requirement → Architecture → Task → Claim → Code → Test → Review → Merge Gate
```

系统需要帮助人类回答以下问题：

```text
需求有没有被设计覆盖？
设计有没有被任务覆盖？
任务有没有被代码实现？
代码有没有被测试验证？
测试有没有覆盖验收标准？
风险有没有被识别？
AI 的判断是否有证据？
```

Vibe Tracing Agent 需要把代码、测试、任务和审查结果，转化为没有开发经验的业务人员和项目管理者能看懂的信息，包括：

```text
流程
数据
状态
输入输出
异常
测试
风险
决策依据
证据来源
门禁结论
```

人类根据这些信息判断：

> 这个系统的逻辑是否成立，边界是否完整，风险是否可控，交付是否可信。

本项目不是一个代码生成工具，而是一个面向 AI Coding 全生命周期的治理、追踪、验收与可视化系统。

---

### 1.4 核心管理哲学

Vibe Tracing 的核心管理哲学是**“黑盒执行，白盒审计；结果导向，过程放权”**。

1. **结果导向，而非微观管理（Result-Oriented vs. Process Micromanagement）**
   我们不需要在 AI Agent 的开发过程中通过极为繁琐细碎的“提示词限制”或“工具（Skill）削减”来面面俱到地管教和约束它。管教过程是脆弱的、极易被绕过、且难以适配大模型的高速演进。
   VT 的核心是在终点设立客观的“交付契约”，通过在收口处对交付结果进行契约检查，并在检测到偏离时及时提供精确指引信息，实现Agent零预设提示词的自适应对齐。

2. **把“规则检查器”变成 Agent 的调试工具（Local Feedback Loop）**
   VT 并不是靠人类来主动督促 Agent。也并非强行限制 AI Agent 的编码自由度，而是在其发生语义漂移、架构越界或测试缺口时，能够精准定位设计与实现之间的偏差（Drifts），并输出包含具体修复动作的指引元数据，引导 Agent 自主修正对齐既定基线。

3. **面向AI Coding Agent友好**
   将审计与报错信息升华为对 Agent 友好的诊断上下文（Diagnostic Metadata）。通过提供清晰的错误根因与修复建议，确保 Agent 能自主解析、自适应补齐依赖并修正逻辑漏洞，在无人工干预下完成自愈。

4. **需求与实现双向对齐**
   承认vibe coding过程中需求的动态演进。VT 通过 finalize（锁定设计基线）与 analyze（开发期审计）两个明确阶段，倒逼 Agent 在修改代码实现的同时，反向更新 PRD、架构约束与任务列表，防止文档与物理代码产生偏离。

5. **第一性原理的硬性追溯链**
   建立从“PRD（需求） → 架构约束 → 任务规划 → 物理代码 → 实证凭证”的确定性追溯链路。拒绝任何隐式推断，任何未申明修改（如 Ghost Code）及无源证据皆判定为偏差，实现黑盒生成物的可解释性。


---

### 1.5 项目必须实现
* 必须 support 非开发背景的用户看得懂。
* 必须用业务语言或管理语言解释代码。
* 对coding Agent的实现结果必须采用独立的Vibe tracing Agent进行采证。
* 必须确保有证据支持的结论才可标记为完成。
* 必须区分推测性判断与事实性结论。
* dashboard必须为单体HTML，可在任意浏览器内直接呈现（内嵌css、内嵌JS等）


### 1.6 非目标范围

本项目当前阶段不做以下内容：

* 不做具体代码和功能实现，代码实现由 Coding Agent 完成。
* 不作为 IDE、代码编辑器或通用开发平台。
* 不自动替代人类完成最终业务决策。
* 不自动批准代码合并。
* 不自动判断业务方向是否正确。
* 不在第一阶段建设复杂平台化能力。
* 不在第一阶段建设完整前端工程。
* 不在第一阶段建设数据库系统。
* 不在第一阶段建设完整 DevOps 平台。
* 不在第一阶段解决所有代码质量问题。

## 2. 用户与使用场景

### 2.1 目标用户

#### 业务人员

业务人员关注 AI Coding 产物是否符合业务目标，而不是关注代码实现细节。

业务人员需要查看：

* 业务需求是否被覆盖
* 业务流程是否完整
* 数据流转是否符合业务逻辑
* 输入输出是否符合业务预期
* 异常场景是否符合真实业务场景
* 异常处理是否符合业务要求
* 人工确认点是否合理
* AI 是否遗漏关键业务边界
* 当前功能是否可以验收

系统对业务人员的表达方式必须使用业务语言、流程语言和逻辑语言，不能要求业务人员理解代码。

#### 项目经理

项目经理负责管理 Vibe Coding 过程中的需求覆盖、任务追踪、实现证据、测试证据、风险暴露和门禁判断。

项目经理需要查看：

* 架构约束是否被遵守
* 需求是否被任务覆盖
* 任务是否有实现证据
* Agent Claim 是否有证据支撑
* 测试是否覆盖验收标准
* 是否存在高风险缺口
* 是否允许进入合并或交付阶段
* 是否存在技术债、异常路径遗漏、不可控依赖等问题

系统对项目经理的表达方式应以管理语言、逻辑语言、风险语言为主，不允许简单地“以代码解释代码”。

#### AI Coding 流程中的 Review / Trace Agent

Review Agent 或 Trace Agent 作为系统内部的治理执行者，需要基于原始材料生成可追踪、可复核、可展示的治理结论。

这类 Agent 需要遵守以下原则：

* 不允许把推测写成事实
* 所有结论必须有证据
* 没找到测试就写 missing
* 看不懂的模块标记为低置信度或不明确
* 不允许 Agent 自证清白
* 高风险项必须给出业务角度解释

### 2.2 核心使用场景

#### SCENE-VT-001：查看需求覆盖情况

* 使用者：项目经理、业务人员
* 触发条件：PRD 已定义，任务列表或实现结果已经产生
* 前置条件：

  * 已存在需求文档
  * 已存在任务拆解结果或 Agent Claim
* 用户操作：

  * 用户打开 Dashboard
  * 查看每个需求是否有关联任务、实现证据、测试证据
* 系统响应：

  * 展示需求覆盖状态
  * 标记已覆盖、部分覆盖、未覆盖、不明确的需求
  * 对缺少证据的需求标记为证据不足或不明确
* 期望结果：

  * 用户可以判断哪些需求已经进入实现链路
  * 用户可以发现未被任务覆盖或未被测试覆盖的需求

---

#### SCENE-VT-002：查看验收标准是否有测试证据

* 使用者：项目经理、业务人员
* 触发条件：某个功能被 Agent 声称已完成
* 前置条件：

  * PRD 中存在验收标准
  * 已存在测试结果或测试文件证据
* 用户操作：

  * 用户查看某个需求下的验收标准矩阵
* 系统响应：

  * 展示每条验收标准对应的测试覆盖情况
  * 标记已覆盖、缺失、不明确
  * 对 Must 级验收标准缺少测试的情况标记为高风险
* 期望结果：

  * 用户可以判断功能是否具备验收依据
  * 用户可以避免仅凭 Agent 声称完成就接受交付

---

#### SCENE-VT-003：查看 Agent Claim 是否有证据支持

* 使用者：项目经理
* 触发条件：Coding Agent 或其他 Agent 声称完成某项任务
* 前置条件：

  * 已存在 Agent Claim
  * 已存在代码、测试、文档或审查结果等证据材料
* 用户操作：

  * 用户查看任务追踪视图
* 系统响应：

  * 展示 Agent Claim 与实际证据之间的对应关系
  * 标记 Claim 是否有需求、任务、代码、测试等证据支持
  * 对证据不足的 Claim 标记为证据不足或不明确
* 期望结果：

  * 用户可以识别 Agent 是否存在无证据自证、遗漏实现、虚假完成等问题

---

#### SCENE-VT-004：查看业务流程图

* 使用者：业务人员、项目经理
* 触发条件：系统完成对项目或模块的分析后
* 前置条件：

  * 已存在 PRD、任务、代码或测试等事实材料
* 用户操作：

  * 用户查看项目级或模块级业务流程图
* 系统响应：

  * 展示业务流程步骤
  * 展示正常路径、异常路径、失败路径、人工确认点
  * 标记流程中缺少证据或无法确认的节点
* 期望结果：

  * 用户可以判断流程是否符合业务逻辑
  * 用户可以识别 happy path 之外是否覆盖异常和边界场景

---

#### SCENE-VT-005：查看输入输出与跨模块数据衔接

* 使用者：业务人员、项目经理
* 触发条件：系统完成模块分析后
* 前置条件：

  * 已存在模块信息、任务信息、代码或测试证据
* 用户操作：

  * 用户查看输入输出表
* 系统响应：

  * 展示模块输入、输出、正常情况、异常情况、失败处理、依赖模块、下游模块
  * 标记字段缺失、字段不一致、上下游无法衔接、隐藏假设等问题
* 期望结果：

  * 用户可以判断模块之间的数据是否能正确流转
  * 用户可以发现“单个函数能跑，但模块之间对不上”的风险

---

#### SCENE-VT-006：查看合并门禁结论

* 使用者：项目经理
* 触发条件：某批任务准备合并或交付
* 前置条件：

  * 已存在需求、任务、Agent Claim、测试结果、审查结果等信息
* 用户操作：

  * 用户查看 Merge Gate 结论
* 系统响应：

  * 输出允许、阻塞、有条件允许等门禁结论
  * 展示阻塞原因
  * 展示必须修复项和可接受风险
* 期望结果：

  * 用户可以基于证据判断是否允许合并
  * 用户可以避免未经验证 of AI 代码进入交付阶段

---

#### SCENE-VT-007：查看风险与后续处理建议

* 使用者：项目经理、业务人员
* 触发条件：系统识别到需求缺口、测试缺口、证据不足、异常场景缺失或架构约束风险
* 前置条件：

  * 已存在分析报告或追踪报告
* 用户操作：

  * 用户查看风险清单
* 系统响应：

  * 展示风险描述
  * 展示风险影响范围
  * 展示证据来源
  * 展示建议处理方式
  * 对高风险项使用业务语言解释影响
* 期望结果：

  * 用户可以理解风险对交付、验收或业务目标的影响
  * 用户可以决定是否修复、接受或重新拆解任务

---

#### SCENE-VT-008：项目配置定型场景

* 使用者：项目经理、架构负责人
* 触发条件：项目已完成架构设计，需要将语言和验证工具配置定型
* 前置条件：

  * 已存在 PRD 和架构约束
  * 架构约束中包含 project.language 和 language_tool_matrix
* 用户操作：

  * 用户运行 `vibe-tracing finalize` 命令
  * VT 从 architecture_constraints.json 读取项目语言和对应的验证工具列表
  * 用户查看 config.json 确认定型结果
* 系统响应：

  * 从架构约束中读取 project.language 和 language_tool_matrix 中该语言的工具列表
  * 将 language 和 validation_tools 写入 .vibetracing/config.json
  * 架构约束中无 project.language 时，通过 stderr 报错并退出
  * config.json 中已有 language 且与架构约束一致时，输出"Already finalized"并跳过
* 期望结果：

  * 项目配置完成定型，后续 analyze 命令可基于 config.json 中的 validation_tools 自行执行验证工具
  * 定型后的 config.json 不再被自动修改

## 3. 功能需求

### REQ-VT-001：全链路需求追踪

#### 需求说明

系统应支持从需求到任务、Agent Claim、代码、测试、审查结果、合并门禁的全链路追踪，使用户能够判断每个需求是否被设计、拆解、实现、测试和审查覆盖。

#### 业务价值

AI Coding 过程中，Agent 可能声称完成任务，但实际可能存在需求遗漏、测试缺失、实现不完整或无证据支撑。全链路追踪可以帮助用户基于证据判断项目进展，而不是依赖 Agent 的口头结论。

#### 类别

functional

#### 优先级

Must

#### 功能范围

* **包含**：

  * 追踪需求是否有关联任务
  * 追踪任务是否有关联 Agent Claim
  * 追踪 Agent Claim 是否有关联代码或测试证据
  * 追踪验收标准是否被测试覆盖
  * 标记缺失、部分覆盖、不明确的链路节点
  * 展示需求、任务、Claim、代码、测试、Review 之间的追踪关系

* **不包含**：

  * 自动编写业务需求
  * 自动替代人类进行业务决策
  * 自动生成最终可上线代码
  * 在 PRD 阶段定义完整追踪数据模型

#### 验收标准

##### AC-VT-001-01：需求必须能关联任务

* 条件：PRD 中存在需求
* 输入：需求文档、任务列表
* 期望输出：系统展示每个需求是否有关联任务
* 异常处理：无法找到关联任务时，必须标记为任务缺失或不明确
* 是否必须有测试：是

##### AC-VT-001-02：验收标准必须能关联测试证据

* 条件：需求中存在验收标准
* 输入：验收标准、测试结果或测试文件证据
* 期望输出：系统展示每条验收标准的测试覆盖状态
* 异常处理：无测试证据时，必须标记为测试缺失，不允许显示为完成
* 是否必须有测试：是

##### AC-VT-001-03：无证据结论不得显示为完成

* 条件：系统生成追踪结论
* 输入：需求、任务、Claim、代码、测试等材料
* 期望输出：没有证据的结论只能显示为不明确、低置信度或证据不足
* 异常处理：如果证据缺失，必须提示证据不足
* 是否必须有测试：是

##### AC-VT-001-04：链路断点必须显式展示

* 条件：需求、任务、Claim、代码、测试之间存在无法连接的节点
* 输入：项目原始材料
* 期望输出：系统展示链路断点和影响范围
* 异常处理：不得静默忽略链路断点
* 是否必须有测试：是

#### 边界场景

* 需求存在，但任务列表为空
* 任务存在，但没有关联需求
* Agent Claim 存在，但没有代码证据
* 测试存在，但无法对应验收标准
* 同一条需求被多个任务部分覆盖
* 同一个任务覆盖多个需求
* 需求 ID 或任务 ID 不规范

#### 异常场景

* 输入文件缺失
* 输入格式无法解析
* 需求 ID 重复
* 任务 ID 重复
* Claim 引用了不存在的需求或任务
* 测试结果无法读取
* 原始材料之间存在冲突

### REQ-VT-002：证据驱动的 Agent Claim 校验

#### 需求说明

系统应校验 Agent 声称完成的任务是否存在实际证据支持，包括需求证据、任务证据、代码证据、测试证据、审查证据等。

#### 业务价值

AI Agent 可能出现过度自信、幻觉、偷懒、遗漏边界等问题。通过证据驱动的 Claim 校验，可以防止”Agent 说完成了”直接等同于”任务真的完成了”。

#### 类别

functional

#### 优先级

Must

#### 功能范围

* **包含**：

  * 读取 Agent Claim
  * 检查 Claim 是否有关联任务
  * 检查 Claim 是否有关联代码、测试或文档证据
  * 标记证据充分、证据不足、无法判断的 Claim
  * 展示 Claim 与 Evidence 的差异
  * 标记 Claim 是否可用于验收判断

* **不包含**：

  * 允许 Agent 无证据自证
  * 自动相信 Claim 结论
  * 自动修改 Claim 内容
  * 在 PRD 阶段定义完整 Agent Claim Schema

#### 验收标准

##### AC-VT-002-01：Claim 必须有证据来源

* 条件：系统读取 Agent Claim
* 输入：Agent Claim 文件
* 期望输出：每个 Claim 都必须展示其证据来源或证据缺失状态
* 异常处理：缺少证据时，必须标记为证据不足或低置信度
* 是否必须有测试：是

##### AC-VT-002-02：Claim 不得自行证明完成

* 条件：Agent Claim 声称任务已完成
* 输入：Claim、任务、代码、测试结果
* 期望输出：系统必须基于外部证据判断 Claim 是否可信
* 异常处理：只有 Claim、没有外部证据时，不得显示为完成
* 是否必须有测试：是

##### AC-VT-002-03：Claim 与证据不一致时必须提示风险

* 条件：Claim 声称完成，但证据只覆盖部分内容或证据冲突
* 输入：Claim 与相关证据
* 期望输出：系统标记 Claim 与证据之间的差异
* 异常处理：不得把部分证据解释为完整完成
* 是否必须有测试：是

##### AC-VT-002-04：同名证据多重匹配与 Fail-Fast 判定

* 条件：多份测试报告中包含相同的 NodeID/路径
* 输入：Claim 与相关证据集合
* 期望输出：系统能够多重匹配并链接该 NodeID 对应的所有证据 ID；若有任意一个测试运行失败，判定该 Claim 为不通过；若结果冲突，在报告中暴露冲突警报
* 是否必须有测试：是

#### 边界场景

* Claim 有任务 ID，但任务不存在
* Claim 有代码路径，但代码路径不存在
* Claim 有测试路径，但测试路径不存在
* Claim 声称完成，但测试失败
* Claim 只覆盖部分验收标准
* Claim 引用了过期文件
* Claim 只包含自然语言说明，没有可验证证据

#### 异常场景

* Claim 文件格式错误
* Claim 缺少 ID
* Claim 缺少关联任务
* Claim 中证据为空
* Claim 与任务目标不一致
* Claim 与测试结果冲突

#### 依赖需求

* REQ-VT-001

### REQ-VT-003：业务流程图生成与检查

#### 需求说明

系统应将项目或模块的实现逻辑转化为业务人员可以理解的业务流程图，并明确展示正常路径、异常路径、失败路径和人工确认点。

#### 业务价值

非开发背景用户无法通过阅读代码判断流程是否符合业务逻辑。业务流程图可以帮助用户检查流程是否完整、顺序是否正确、分支是否清晰、异常处理是否符合预期。

#### 类别

functional

#### 优先级

Should

#### 功能范围

* **包含**：

  * 输出项目级业务流程图

  * 展示正常流程
  * 展示异常流程
  * 展示失败处理
  * 展示人工确认点
  * 标记缺少证据的流程节点

* **不包含**：

  * 要求用户阅读代码
  * 用代码调用关系替代业务流程
  * 自动决定业务流程是否最终正确
  * 第一阶段强制实现完整流程图能力

#### 验收标准

##### AC-VT-003-01：必须输出业务语言流程图

* 条件：系统完成项目或模块分析
* 输入：需求、任务、代码、测试或文档证据
* 期望输出：输出业务人员可理解的流程图
* 异常处理：无法确认的流程节点必须标记为不明确
* 是否必须有测试：是

##### AC-VT-003-02：流程图应包含异常路径

* 条件：功能存在异常场景或边界场景
* 输入：PRD、测试结果、代码证据
* 期望输出：流程图中展示成功、失败、空数据、重复、冲突等路径
* 异常处理：异常路径缺失时标记为风险
* 是否必须有测试：是

##### AC-VT-003-03：流程图必须能够关联证据

* 条件：流程图展示某个流程节点
* 输入：流程节点、需求、任务、代码、测试等材料
* 期望输出：流程节点可以追溯到对应证据或标记为证据不足
* 异常处理：无证据节点不得标记为已确认
* 是否必须有测试：是

#### 边界场景

* 只有正常流程，没有异常流程
* 代码存在异常处理，但 PRD 未定义异常
* PRD 定义异常，但实现中无法找到证据
* 模块流程与项目流程不一致
* 流程中存在人工确认点但未明确责任人

#### 异常场景

* 无法识别流程
* 流程节点无证据支撑
* 流程顺序与 PRD 不一致
* 流程中存在隐藏假设
* 流程图过度技术化，业务人员无法理解

* REQ-VT-001
* REQ-VT-006

### REQ-VT-004：输入输出表与跨模块衔接检查

#### 需求说明

系统应支持按照项目级和模块级输出输入输出表，帮助用户查看每个模块需要什么输入、产生什么输出、异常时如何处理、依赖哪些模块、影响哪些下游模块。

#### 业务价值

AI Coding 可能出现单个模块看似可用，但上下游字段不一致、数据结构不匹配、隐藏假设未暴露等问题。输入输出表可以帮助用户从业务和数据流角度判断系统是否真正可运行。

#### 类别

functional

#### 优先级

Should

#### 功能范围

* **包含**：

  * 展示模块输入
  * 展示模块输出
  * 展示正常情况
  * 展示异常情况
  * 展示失败处理
  * 展示依赖模块
  * 展示下游模块
  * 检查跨模块输入输出是否匹配

* **不包含**：

  * 只展示函数参数，不解释业务含义
  * 只展示代码结构，不说明数据流转
  * 自动修改模块接口
  * 第一阶段强制完成所有跨模块语义分析

* 期望输出：展示输入、输出、异常、依赖、下游模块
* 异常处理：无法确认字段来源时，标记为不明确
* 是否必须有测试：是

##### AC-VT-004-02：必须检查上下游字段是否匹配

* 条件：存在模块依赖关系
* 输入：上游输出、下游输入
* 期望输出：标记匹配、不匹配、不明确
* 异常处理：字段不一致时标记为集成风险
* 是否必须有测试：是

##### AC-VT-004-03：隐藏输入假设必须暴露

* 条件：模块依赖默认输入格式、默认编码、默认字段或默认状态

* 是否必须有测试：否

* 上游输出字段缺失
* 下游依赖未声明字段
* 字段名相似但语义不同
* 默认输入假设未明示
* 异常输出未定义
* 多个下游模块依赖同一个输出

#### 异常场景

* 无法识别模块输入
* 无法识别模块输出
* 下游模块不存在
* 模块依赖循环
* 输入输出表缺少业务含义
* 输出格式与测试期望不一致

### REQ-VT-006：Dashboard 治理视图

#### 需求说明

系统应提供面向人类阅读的 Dashboard，用于展示需求覆盖、业务流程、输入输出、数据结构、测试矩阵、异常场景、风险和合并门禁结论。

#### 业务价值

人类不应该在多个代码文件、测试文件、Agent 输出文件之间手工查找证据。Dashboard 应作为统一治理入口，让业务人员和项目经理快速判断项目状态、风险和下一步行动。

#### 类别

functional

#### 优先级

Must

#### 功能范围

* **包含**：

  * 展示模块状态
  * 展示需求覆盖情况
  * 展示验收标准测试覆盖情况
  * 展示 Agent Claim 证据情况
  * 展示业务流程图
  * 展示输入输出表
  * 展示数据结构
  * 展示测试矩阵
  * 展示异常场景
  * 展示风险清单
  * 展示合并门禁结论
  * 展示必须修复项和建议修复项

* **不包含**：

  * 在 Dashboard 中重新推理结论
  * 用 Dashboard 替代原始证据文件

  * 在第一阶段实现完整交互式项目管理系统

##### AC-VT-006-01：Dashboard 必须展示核心治理维度

##### AC-VT-006-02：Dashboard 不得重新解释事实

* 输入：分析报告
* 期望输出：Dashboard 只展示报告中的结论 and 证据
* 异常处理：报告缺失时，不得生成虚假结论
* 是否必须有测试：是

##### AC-VT-006-03：Dashboard 必须面向非开发用户可读

* 条件：Dashboard 展示风险、缺口或门禁结论
* 输入：分析报告
* 期望输出：使用业务语言、管理语言解释问题
* 异常处理：不得只输出代码路径或技术术语而不解释业务影响
* 是否必须有测试：否

##### AC-VT-006-04：Dashboard 必须完全使用矢量图形 (SVG) 代替所有 Emoji 表情

* 条件：Dashboard 渲染导航菜单、状态徽章、警报组件等界面元素

* 某风险没有处理建议

#### 异常场景

* Dashboard 输入文件不存在
* Dashboard 输入格式错误
* Dashboard 中存在无法渲染的数据

### REQ-VT-007：风险、反驳项与技术债识别

#### 需求说明

系统应识别需求、任务、实现、测试、架构约束和 Agent Claim 中的风险，并在关键节点提供反驳项，避免 AI Agent 过度迎合人类或只输出乐观结论。

#### 业务价值

AI Agent 容易为了完成任务而忽略风险、异常、边界和代价。系统需要主动暴露风险、替代方案和适用条件，帮助人类做更稳健的管理决策。

#### 类别

functional

#### 优先级

Should

#### 功能范围

* **包含**：

  * 输出风险清单
  * 输出技术债清单
  * 输出当前方案风险
  * 输出是否存在更优方案
  * 输出替代方案适用条件
  * 用业务语言解释高风险项
  * 标记需要人类确认的事项

* **不包含**：

  * 自动替人类选择最终方案
  * 只输出迎合性结论
  * 将推测性风险标记为事实
  * 在第一阶段实现完整风险评分模型

#### 验收标准

##### AC-VT-007-01：关键结论必须包含风险说明

* 条件：系统输出需求、架构、任务、验收或门禁结论
* 输入：分析报告
* 期望输出：结论中包含风险、证据、影响和建议
* 异常处理：无法判断风险时，标记为不明确
* 是否必须有测试：是

##### AC-VT-007-02：高风险项必须用业务语言解释

* 条件：系统识别到高风险项
* 输入：风险证据
* 期望输出：解释该风险对业务、验收或交付的影响
* 异常处理：缺少证据时不得标记为已确认风险
* 是否必须有测试：是

##### AC-VT-007-03：系统应提供必要反驳项

* 条件：人类或 Agent 提出关键方案、关键判断或关键交付声明
* 输入：方案、Claim、证据材料
* 期望输出：系统提示当前方案风险、替代方案、采用条件和代价
* 异常处理：无证据支持的反驳项应标记为建议或推测，不得标记为事实
* 是否必须有测试：否

#### 边界场景

* 风险只有推测，没有证据
* 风险对技术影响大，但业务影响小
* 风险短期可接受，长期不可接受
* Agent 只给出方案优点，没有说明代价
* 技术债存在但不阻塞当前交付

#### 异常场景

* 高风险项未展示
* 风险没有来源

### REQ-VT-008：合并门禁判断

#### 需求说明

系统应根据需求覆盖、验收标准测试覆盖、Agent Claim 证据、架构约束违反情况和风险等级，生成合并门禁结论。

#### 业务价值

AI Coding 的交付不能只依赖 Agent 声称完成。合并门禁可以帮助项目经理判断当前结果是否允许进入下一阶段，是否必须阻塞，或者是否可以带风险接受。

#### 类别

functional

#### 优先级

Must

#### 功能范围

* **包含**：

  * 判断是否允许合并

  * 展示必须修复项
  * 展示可接受风险
  * 对缺少证据的结论进行降级
  * 说明门禁判断依据

* **不包含**：

  * 自动合并代码
  * 自动忽略高风险项
  * 自动将低置信度内容标记为完成
  * 在 PRD 阶段定义完整门禁算法

#### 验收标准

##### AC-VT-008-01：Must 级验收标准缺少测试时必须阻塞

* 条件：存在 Must 级需求或验收标准
* 输入：验收标准、测试覆盖状态
* 期望输出：缺少测试证据时，Merge Gate 输出阻塞结论
* 异常处理：证据不明确时，输出阻塞或有条件阻塞
* 是否必须有测试：是

##### AC-VT-008-02：高风险无处理建议时必须阻塞

* 条件：存在高风险项
* 输入：风险清单、处理建议
* 期望输出：没有处理建议或责任归属时，Merge Gate 输出阻塞结论
* 异常处理：风险证据不足时，标记为不明确，不得直接允许
* 是否必须有测试：是

##### AC-VT-008-03：门禁结论必须说明依据

* 条件：系统输出门禁结论

* 是否必须有测试：是

* 测试通过，但需求没有任务覆盖
* Claim 有证据，但证据只覆盖部分功能
* 架构约束状态不明确
* 风险存在但业务上可接受
* 部分功能允许合并，部分功能需要阻塞

#### 异常场景

* 无法生成门禁结论

### REQ-VT-009：项目生命周期管理与工具执行验证

#### 需求说明

系统应支持项目从前期设计到开发验证的完整生命周期管理，包括项目配置定型（finalize）、VT 自主执行验证工具、Claim 可信度判定。VT 通过自行执行白名单中的验证工具获取高可信度证据，而非依赖 Agent 放置的报告文件。

#### 业务价值

Vibe Tracing 的目标是治理 AI Coding 全生命周期。通过项目配置定型和工具执行验证，确保 VT 能够基于自身执行的工具结果判定 Claim 可信度，形成”VT 验证 VT”的客观闭环，而非依赖 Agent 自报的完成状态。

#### 类别

functional

#### 优先级

Must

#### 功能范围

* **包含**：

  * 支持项目配置定型，从架构约束中读取项目语言和验证工具写入 config.json
  * 支持 VT 根据 config.json 中的 validation_tools 和 language_tool_matrix 自行执行验证工具
  * 支持 Claim 可信度判定，无 VT 执行的工具证据时标记为 low_confidence
  * 要求架构约束必须声明项目语言和可用验证工具（language_tool_matrix）
  * 要求工具执行失败时通过 stderr 输出精确错误信息和修复建议
  * 要求与治理相关的 Agent 输出进入结构化 JSON 治理文件

* **不包含**：

  * 不读取 Agent 放置的 .vibetracing/tool_reports/ 目录文件
  * 不执行 language_tool_matrix 白名单之外的任意命令
  * 不允许 Agent 自报的完成状态作为门禁判定的充分依据
  * 不在 PRD 阶段定义完整工具执行引擎实现细节

##### AC-VT-009-01：MVP 必须明确 Claude Code 首选执行环境

##### AC-VT-009-03：subagent 职责和 skill 使用必须可审查

* 输入：子智能体运行历史
* 期望输出：职责和 skill 可审查
* 是否必须有测试：是

##### AC-VT-009-04：架构约束必须声明项目语言和可用验证工具

* 条件：项目使用 Vibe Tracing 进行治理
* 输入：architecture_constraints.json
* 期望输出：架构约束中必须包含 project.language 字段和 language_tool_matrix 顶层字段，language_tool_matrix 定义每种编程语言可用的验证工具、默认命令模板、输出格式和通过条件

##### AC-VT-009-05：目录结构配置化与引擎解耦

* 异常处理：直接编码而未经规程流转或文档未同步时，门禁校验应能抛出规程违反警告/错误

* 异常处理：数据项字段校验失败但 Schema 中无对应 `description` 时，应正常输出常规错误信息。

##### AC-VT-009-09：VT 必须能自行执行 Claim 关联的验证工具

* 输入：config.json 中的 validation_tools、架构约束中的 language_tool_matrix 命令模板、agent_claims.json 中的 Claim 关联测试路径
* 期望输出：
  1. VT 根据 validation_tools 列表，从 language_tool_matrix 取命令模板。
  2. 替换路径占位符后执行命令，捕获 stdout/stderr。
  3. 解析工具输出，生成证据条目写入 evidence_index.json。
  4. 不读取 Agent 放置的 .vibetracing/tool_reports/ 目录文件。
* 异常处理：工具未安装时 stderr 输出安装建议；测试路径不存在时 stderr 输出具体路径。
* 是否必须有测试：是

##### AC-VT-009-10：工具执行失败时必须精确反馈

  4. 配置缺失：stderr 输出"Error: config.json missing 'language' field. Run 'vibe-tracing finalize' first."。
* 异常处理：错误信息必须包含具体原因和修复建议，不得输出通用错误。

* 异常处理：无。
* 是否必须有测试：是

* 输入：同时存在 AC gap（blocked 级）和 REQ gap（fail 级）
* 期望输出：
  1. Gate Decision 为 blocked（最高优先级）。
  2. reasons 列表同时包含 AC gap 和 REQ gap 的描述。

* 输入：被修改的 prd.md
* 期望输出：
  1. vt finalize 必须同时存储 constraints_hash 和 prd_hash 到 config.json。
  2. vt analyze 检测到 PRD 哈希不匹配时输出 WARNING（不阻断）。
  3. vt analyze 检测到 PRD↔Architecture 映射失效时必须阻断。
* 异常处理：config.json 中无 prd_hash 字段时跳过检测。
* 是否必须有测试：是

##### AC-VT-009-15：PRD↔Architecture 映射必须在 analyze 时持续校验

* 条件：执行 vibe-tracing analyze 且 PRD 或 constraints 存在
* 输入：当前 PRD + 当前 constraints
* 期望输出：
  1. 死链检测：constraints 引用的 REQ 不存在于 PRD → 阻断。
  2. MUST 覆盖检测：MUST 级 REQ 无架构支撑 → 阻断。

* 条件：执行 vibe-tracing analyze 且存在 completed Claim
* 输入：Claim 的 test_refs + 关联 task 的 AC 列表

* 是否必须有测试：是

##### AC-VT-009-17：Pre-commit hook 必须可靠且快速

* 条件：vt init 安装 pre-commit hook 后执行 git commit
* 输入：staged 文件
* 期望输出：

### REQ-VT-010：质量演进生命周期管理

#### 业务价值

VT 自身也由 AI Agent 开发，需要一套自我诊断与持续改进机制。通过 8 维度反思诊断，VT 能够主动识别自身的技术债、架构偏差和质量缺口，并产出可执行的进化计划，确保框架自身的可持续演进。

##### AC-VT-010-02：反思诊断覆盖校验

* 条件：反思诊断输出包含 Affected Scope
* 输入：反思诊断结果、task_list.json 的 code_refs
* 期望输出：与 task_list.json 的 code_refs 做覆盖校验，未覆盖文件输出 WARNING
* 异常处理：task_list.json 不存在或格式错误时，跳过覆盖校验并输出提示
* 是否必须有测试：是

##### AC-VT-010-03：进化计划结构化输出

* 条件：vt analyze 完成且存在质量缺陷
* 输入：反思诊断结果、质量缺陷列表
* 期望输出：可生成结构化自我进化计划
* 异常处理：无质量缺陷时，输出"无需进化"并正常结束
* 是否必须有测试：是

## 4. 非功能需求

### 性能

* 单次项目分析应在用户可接受时间内给出进度反馈。
* 第一阶段不追求大型企业级性能，重点保证小型到中型代码库可以完成分析。
* Dashboard 打开后应能快速展示核心摘要信息。
* 对于耗时较长的分析过程，应至少提供状态信息，避免用户无法判断系统是否卡住。
* 系统应避免对相同材料进行无意义重复分析。
* 后续架构设计阶段应考虑模型调用成本和上下文复用能力。

### 容量

* 第一阶段面向单项目代码库。
* 系统应支持读取 PRD、架构约束、任务列表、Agent Claim、代码库、测试结果等材料。
* 第一阶段不建设复杂数据库，优先使用文件化输入输出。
* 输出报告应可被长期保存和重复查看。
* 系统应能够处理持续增长的任务、Claim、测试结果和 Review 记录。
* 对超出第一阶段容量预期的项目，应给出分析范围说明。

---

### 安全

* 系统不得将无证据结论标记为事实。
* 系统不得让同一个 Agent 对自己的任务进行最终验收。
* 系统应避免在报告中泄露敏感信息。
* 如分析结果包含路径、代码片段或日志，应避免暴露不必要的密钥、Token、账号等敏感数据。
* 基础安全风险应能够被识别并展示为风险项。
* 系统应避免执行不可信输入中的危险指令。
* 系统应明确区分事实、推断和建议。

---

### 可靠性

* 输入文件缺失、格式错误、字段不完整时，系统应给出明确错误信息。
* 系统不得因为部分数据缺失而生成虚假完整结论。
* 分析无法确认时，应输出不明确或低置信度结论。
* Dashboard 必须展示缺失项，而不是静默忽略。
* 关键结论必须可追溯到证据来源。
* 报告生成失败时，应保留可诊断信息。
* 系统应支持重复运行，并尽量保证相同输入下产生一致结果。

---

### 兼容性

* PRD 可使用 Markdown。
* 面向机器传递的结构化信息应优先使用 JSON。
* 面向人类阅读的结果应通过 HTML Dashboard 展示。
* 第一阶段应优先支持本地项目文件结构。
* 第一阶段不要求支持复杂前端工程。
* 第一阶段不要求支持数据库服务。
* 后续可扩展支持更多语言、工具链和项目结构。

---

### 可观测性

* 核心流程必须有日志。
* 核心分析结果必须可视化。
* 错误、缺失、低置信度、高风险项必须显式展示。
* 系统应建立清晰的错误码和日志体系。
* 用户应能看到哪些输入被使用，哪些证据支撑了哪些结论。
* 用户应能看到分析过程中哪些部分无法判断。
* 系统应能展示门禁结论的来源和理由。

---

### 可维护性

* 系统应尽量保持输入层、分析层、报告层、展示层职责清晰。
* 人类写作内容可以是 Markdown。
* 机器之间传递内容应尽量是 JSON。
* 人类阅读内容应以 HTML Dashboard 为主。
* 规则和约束应尽量结构化，便于后续扩展。
* 架构设计阶段应进一步定义核心对象模型、状态枚举、证据模型和数据契约。
* 系统应避免把业务需求、架构设计、实现细节混在同一层。


---

### 边界条件

* 空 PRD：系统应提示缺少需求输入。
* 空任务列表：所有需求应标记为任务缺失或不明确。
* 空 Claim：任务完成状态不得被标记为完成。
* 空测试结果：验收标准测试状态应标记为测试缺失。
* 空 Review 结果：不得假设 Review 已通过。
* 超大代码库：系统应给出分析范围说明。
* 网络中断：本地分析流程不应依赖持续联网。
* 恶意输入：系统应避免执行不可信输入中的危险指令。
* 不完整 evidence：结论应降级为证据不足或低置信度。
* 输入材料之间冲突：系统应显式展示冲突，不得自行选择性忽略。

---

## 5. 验收总则

一个需求只有同时满足以下条件，才可认为完成：

* 已有关联任务
* 已有 Agent Claim
* 已有代码实现证据
* 已有测试证据
* Must 级验收标准必须有测试覆盖
* 已覆盖异常场景
* 已覆盖边界场景
* 未违反架构约束
* 高风险项已被识别并有处理建议
* Review Agent 未发现阻塞问题
* Merge Gate 结论不是阻塞
* 所有完成结论均有 evidence 支撑

以下情况不得标记为 Done：

* 只有 Agent Claim，没有外部证据
* 只有代码，没有测试
* 只有测试结果，但无法对应验收标准
* 只有推测，没有 evidence
* 关键输入文件缺失
* 需求、任务、代码、测试之间无法建立追踪关系
* 架构约束状态不明确但被假设为合规
* 高风险项未解释或未处理
* Review 结果缺失但被假设为通过

系统必须遵守以下验收原则：

1. **Evidence First**
   没有证据，不得标记为完成。

2. **Human Readable**
   所有结论必须让非开发背景用户能理解。

3. **Traceable**
   需求、任务、代码、测试、风险之间必须可追踪。

4. **Conservative Judgment**
   不确定时输出不明确，而不是假装完成。

5. **Agent Cannot Self-Prove**
   Agent Claim 不能作为自身完成的唯一证据。

6. **Markdown for Human, JSON for Machine, HTML for Review**
   人写的内容可以是 Markdown，机器之间传递的内容尽量 JSON，人类读的内容应通过 HTML 展示。

7. **First Version Focuses on Governance, Not Full Code Understanding**
   第一版重点是治理闭环，不追求完整自动理解全部代码。

## 8. 第一阶段范围

### 8.1 第一阶段目标

第一版只回答 4 个核心问题：

```text
1. 每个需求是否有任务覆盖？
2. 每个验收标准是否有测试覆盖？
3. 每个任务的 Agent Claim 是否有证据支持？
4. 是否违反架构约束？
```

第一阶段的目标不是完整理解代码，而是建立最小可用的 AI Coding 治理闭环。

---

### 8.2 第一阶段核心功能

第一阶段必须优先实现：

* REQ-VT-001：全链路需求追踪
* REQ-VT-002：证据驱动的 Agent Claim 校验
* REQ-VT-006：Dashboard 治理视图
* REQ-VT-008：合并门禁判断
* REQ-VT-009：项目生命周期管理与工具执行验证

第一阶段可部分支持：

* REQ-VT-007：风险、反驳项与技术债识别

第一阶段暂不强制完整实现：

* REQ-VT-003：业务流程图生成与检查
* REQ-VT-004：输入输出表与跨模块衔接检查
* REQ-VT-005：数据结构列表与业务解释

---

### 8.3 第一阶段输入

第一阶段建议至少支持以下输入：

```text
prd.md
architecture_constraints.json
task_list.json
agent_claims.json
代码库
测试结果摘要
Review 结果摘要（可选）
```

其中：

* PRD 可以是 Markdown
* 架构约束应是结构化文件
* 任务列表应是结构化文件
* Agent Claim 应是结构化文件
* 测试结果可以来自测试工具输出或结构化摘要
* Review 结果可以作为可选输入

---

### 8.4 第一阶段输出

第一阶段至少输出：

```text
traceability_report.json
evidence_index.json
dashboard.html
merge_gate 结论
```

其中：

* traceability_report.json 用于机器读取和后续追踪
* dashboard.html 用于人类查看
* merge_gate 结论用于项目经理判断是否允许进入下一阶段

---

### 8.5 第一阶段成功标准

第一阶段成功标准如下：

* 用户可以看到每个需求是否有任务覆盖。
* 用户可以看到每个验收标准是否有测试覆盖。
* 用户可以看到每个 Agent Claim 是否有证据支持。
* 用户可以看到架构约束是否存在违反或不明确。
* 用户可以看到 VT 工具执行结果和 Claim 可信度判定。
* 用户可以看到阻塞项、风险项和建议动作。
* 系统不会把没有证据的结论标记为完成。
* Dashboard 能让非开发背景用户理解当前项目状态。
* Merge Gate 能给出允许、阻塞或有条件允许的判断。

## 12. 附录：真实性保障原则

系统必须遵守以下真实性保障原则：

```text
1. 不允许把“推测”写成“事实”
2. 所有结论必须带 evidence
3. 没找到测试就写 missing，不允许编造
4. 看不懂的模块标记为不明确或低置信度
5. 业务流程必须能对应到代码、测试或文档
6. Dashboard 只渲染报告，不允许重新解释事实
7. 结构化数据必须能通过后续定义的 Schema 校验
8. 高风险项必须给出业务角度的解释
9. Agent Claim 不能作为自身完成的唯一证据
10. 没有 evidence 的结论不得显示为 Done
```

## 13. 附录：推荐工作流

Vibe Tracing 建议嵌入以下 AI Coding 工作流：

```text
1. 人类定义业务目标，写 PRD
2. 架构 Agent 输出架构约束（含 project.language 和 language_tool_matrix）
3. 运行 vibe-tracing finalize，从架构约束定型项目语言和验证工具到 config.json
4. 任务 Agent 拆解任务列表
5. Test Agent 写测试并跑测试
6. Coding Agent 按任务写代码
7. Review Agent 审查代码
8. Doc Agent 生成非代码交付物
9. 运行 vibe-tracing analyze，VT 自行执行验证工具并生成证据
10. Trace Agent 追踪需求、任务、Claim、代码、测试、Review
11. Dashboard Agent 生成人类可读 Dashboard
12. 人类根据可视化图表、表格、测试报告进行查看与业务方向指导
13. 新的任务进入 PRD 与 task_list，形成下一轮治理闭环
```

人类在该流程中的职责不是直接指导代码实现，而是：

* 判断业务目标是否正确
* 判断流程是否符合业务
* 判断风险是否可接受
* 判断缺口是否需要补任务
* 判断是否允许进入下一阶段

## 14. 附录：项目定位总结

Vibe Tracing 的定位是：

> 一个面向 AI Coding 的证据链追踪与验收治理工具。

它不是：

> 一个自动理解代码并生成所有项目管理文档的超级 Agent。

第一版最重要的目标是让项目经理不用读代码，也能知道：

```text
哪些需求有任务，
哪些验收标准有测试，
哪些 Agent Claim 有证据，
哪些地方必须阻塞合并。
```
