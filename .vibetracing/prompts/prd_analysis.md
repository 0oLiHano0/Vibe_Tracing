# 一、分析原则

## 1. 区分信息来源

你在分析时，必须明确区分以下四类内容：

* 【用户已明确提供】：PRD 原文中已经明确写出的内容；
* 【基于上下文推断】：你根据上下文合理推断出的内容；
* 【建议补充】：你认为业务上应该考虑，但原文没有写出的内容；
* 【需要确认】：不能直接作为开发依据，必须由业务人员确认的内容。

除非 PRD 原文已经明确写出，否则不要把你的推断和建议当成正式需求。

---

## 2. 控制需求膨胀

当你补充遗漏需求时，必须按以下分类输出：

* 【当前版本建议加入】：与当前目标强相关，建议本版本纳入；
* 【后续版本可考虑】：有价值，但不建议当前版本实现；
* 【风险提醒】：仅提示风险，不建议直接转为需求。

不要因为某个功能“常见”或“专业”就默认加入当前版本。

---

## 3. 不做架构设计

你可以指出哪些需求会影响后续架构设计，但不要直接给出技术实现方案。

例如：

* 可以说：该需求可能影响权限设计、数据存储、性能约束；
* 不要说：应该使用 Redis、PostgreSQL、微服务、消息队列、JWT、Docker 等具体技术方案。

具体技术选型、模块划分、接口契约、存储方案、部署方案、日志格式和测试策略，将由后续架构设计环节处理。

---

## 4. 保护原始 PRD

不要直接覆盖或重写我的原始 PRD。

如果你认为某部分需要修改、补充或删除，请以“建议变更”的形式输出，并说明原因。

# 五、第四遍：验收标准完善

请检查每个需求的验收标准是否完整、可验证、可测试。

对每个验收标准，补充或完善以下内容：

* 条件：什么情况下触发验收；
* 输入：给系统什么数据或用户执行什么操作；
* 期望输出：系统应该产生什么业务结果；
* 异常处理：出错或不满足条件时应该怎么办；
* 是否必须有测试：是 / 否；
* 测试建议：人工测试 / 自动化测试 / 单元测试 / 集成测试 / E2E 测试 / 不确定。

* Must 级需求的核心验收标准必须有测试；

* 不要把技术实现细节写进验收标准；
* 验收标准应描述业务可见结果。

# 十、工作流与后续行动指示 (Workflow & Next Actions)

你的最终目标是配合人类用户将 PRD 需求整理得清晰、合规，并推进到可开发状态。请遵循以下协作工作流：

1. **第一阶段：需求分析与澄清（当前阶段）**
   - 严格按照上述“七遍分析法”对 PRD 初稿进行分析，并在对话中输出分析报告。
   - 重点列出所有【需要确认】的问题，等待用户解答。**在用户未作答复前，请勿直接修改 docs/prd.md。**

2. **第二阶段：合意更新与文档沉淀**
   - 在用户答复了你提出的问题后，你应当：
     a. **更新 docs/prd.md**：将确认后的业务场景、功能细节、AC 描述写回并替换 `docs/prd.md` 中的对应占位或草稿内容，将文件头部元属性修改为 `当前状态：active`（或在需求未全部定稿时保持 `draft`）。
     b. **生成任务与契约文件**：
        - 自动在 `docs/task_list.json` 中生成符合规范的任务列表，每个任务要分配唯一的 `TASK-VT-xxx` ID，并与其负责实现的 `REQ`/`AC` 进行显式关联绑定。
        - 自动初始化空数组 `[]` 或模板项至 `.vibetracing/agent_claims.json`。
