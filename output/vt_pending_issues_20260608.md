# VT 待研究问题

本文档记录已识别但尚未解决的结构性问题。这些问题需要独立研究和规划，不纳入当前进化计划。

---

## 一、 测试覆盖率与测试用例分布失衡

### 现象
- 500+ 测试用例，代码行覆盖率仅 42%（低于 80% 阈值）
- 覆盖率工具将 41 个证据标记为 "violated"
- 测试用例数量与覆盖率严重不匹配

### 可能原因
- 大量测试集中在少量模块（如 cli.py、test_helpers），而大量模块几乎无测试
- 测试用例随功能增长自然积累，未有系统的覆盖率规划
- Mock 测试占比过高，实际代码执行路径未被覆盖

### 影响
- 覆盖率 violated 是 hook BLOCKED 的主要来源之一
- 人类无法通过覆盖率数据判断代码质量的真实状态
- 测试投入（500+ 用例）与产出（42% 覆盖率）不成比例

### 需要研究
- 各模块的覆盖率分布热力图
- 测试用例的类型分布（unit/integration/e2e/mock）
- 覆盖率缺口的优先级排序
- 测试策略调整方案

## 二、 噪音消除措施对证据链完整性的影响

### 已实施的噪音消除措施

| 措施 | 消除的噪音 | 损失的信息 |
|---|---|---|
| `accepted_by` 机制 | manual 规则不再输出提示 | 人类确认 ≠ 持续合规，确认后规则从所有输出消失 |
| GATE-VT-002~012 改为 manual | 无检查器的 machine 规则不再报 unclear | VT 自动验证能力边界变得不可见 |
| exit code 5/2 不产生证据 | pytest "no tests collected" 不报错 | "这个文件没有测试"本身是有价值信息 |
| stale gaps/risks | 未变更文件的问题不影响门禁 | 依赖关系变化可能导致旧 gap 回归 |
| 非代码文件不执行工具 | .md/.json 不触发 ruff/mypy | 合理边界，无损失 |

### 核心矛盾

噪音消除和证据链完整性是零和博弈。当前措施偏向"减少 Agent 摩擦"，但以"降低人类可见性"为代价。

### 正确方向：决策平台

从第一性原则出发，人类决策用的是逻辑思维而非阅读代码。当前噪音消除的实质是**该人类决策的地方，没有给人类接手的窗口**。正确的方向不是"分层呈现"，而是"让人类能通过页面作出决策"。

**死循环**：Agent 产生噪音 → 噪音被消除 → 人类看不到全貌 → 人类无法决策 → Agent 继续推进 → 更多待决策 → 更多消除 → 死循环

**打破点**：在"人类无法决策"这个环节，不是减少信息，而是降低决策门槛——dashboard 改造为具备交互功能的决策页面。

- **Agent 视图**：保持精简，快速门禁决策，已决策事项自动排除
- **人类视图**：Dashboard 上的决策卡片，用业务语言描述待决策事项，人类点击按钮完成决策
- **决策记录**：`human_decisions.json` 记录所有人类决策，Agent 读取并应用

> [!NOTE]
> 详细方案见 [噪音消除措施改进方案](vt_noise_reduction_improvement_plan_20260608.md)

### 需要研究
- Dashboard 决策请求展示（第一步，低成本验证）
- 交互式 Dashboard + Decision API（第二步，完整实现）
- `human_decisions.json` 数据结构设计
- Agent 读取决策日志并自动排除已决策噪音的机制

## 三、 关联

这两个问题相互关联：
- 覆盖率低 → 覆盖率 violated 多 → hook BLOCKED → Agent 用 --no-verify 绕过 → 证据链断裂
- 噪音消除 → 真实问题被掩盖 → 人类无法判断代码质量 → 决策迟缓 → Agent 死循环

解决路径：
1. 先解决覆盖率问题（减少真实 BLOCKED）
2. 将噪音消除改造为决策请求（不是隐藏信息，而是让人类决策）
3. Dashboard 改造为决策平台（人类通过页面按钮完成决策）
4. Agent 读取决策日志，已决策事项自动排除噪音
